# Documentation Content Rules

Writing rules for every page on the docs site and every component doc. Derived from the voice observed in the Figma file's own strings. Follow these exactly.

## Voice

- Plain, direct, helpful. Speak to the reader as you and your.
- Confident but never hypey. Concrete claims over adjectives. No jokes in documentation.
- Short declarative sentences. Cut every word that does not add meaning.

## Mechanics

- Sentence case everywhere: titles, headings, buttons, nav labels. No Title Case, no ALL CAPS except tiny eyebrow labels.
- No em dashes. Use a period or a comma and restructure the sentence.
- No emoji and no unicode symbols. Meaning is carried by icons from the Iconography library.
- Button labels are short and verb first: Save changes, Delete, Upgrade to Premium. Destructive actions name the object: Delete listing, never OK.
- Numbers and money follow Indian conventions where they appear: rupee with lakh and crore, Indian digit grouping, sqft for area.

## Structure

- Component pages follow the RULEBOOK section 13 order exactly: Overview, Purpose, When to use, When not to use, Anatomy, Properties, Variants, States, Behaviors, Accessibility, Specs, Do's and don'ts.
- Specs reference token names, never raw values alone. A raw px value may appear only next to its token name.
- If a section genuinely does not apply, say so in one sentence. Never omit it silently.
