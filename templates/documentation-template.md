# ComponentName

Minimal, plain writing. No em dashes. Short declarative sentences. Fill every section, in this order (RULEBOOK §13).

## Overview

One or two sentences. What the component is.

## Purpose

The problem it solves.

## When to use

Valid use cases, as a short list.

## When not to use

Cases where a different component or pattern applies instead. Name the alternative.

## Anatomy

Container, then content areas, then support elements, in hierarchy order. State the layout mode and sizing for each (Auto Layout direction, Hug or Fill or Fixed).

## Properties

Table: name, type (Boolean, Variant, Text, Instance swap), accepted values, default.

| Property | Type | Values | Default |
|---|---|---|---|
| | | | |

## Variants

The full matrix across all variant dimensions. State the dimension count and total combinations.

## States

Every supported interaction state, listed. If a state does not apply, say so and why, rather than omitting silently.

## Behaviors

Interaction, motion, or responsive behavior not evident from static properties. If none, say none.

## Accessibility

Component-specific requirements: minimum tap target, minimum text size, ARIA role, keyboard pattern. Confirm color is never the only carrier of meaning.

## Specs

Redline measurements, referencing token names, not raw values: padding, gap, sizing, radius, border. State resizing behavior and constraints.

## Do's and Don'ts

At least two correct and two incorrect usage examples.
